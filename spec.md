# Ubuntu RD Gateway 仕様書

## 1. 目的
- Windows RD Gateway 相当の機能を Ubuntu 上で実現する。
- プロキシ環境から外部へアクセスするため、RDP 通信を HTTPS でカプセル化する。

## 2. 背景
- 社内ネットワークは HTTP/HTTPS プロキシを経由して外部アクセスする。
- 既存の Windows RD Gateway と同等の接続性を Linux 環境で提供したい。

## 3. 対象範囲
- Ubuntu サーバー上に RDP over HTTPS を提供するゲートウェイ機能を構築する。
- クライアントは標準の RDP クライアントを利用する。
- Windows のネイティブな `mstsc.exe` で接続できることを前提とする。
- RD の対象 PC は Ubuntu とする。

## 4. 要件
### 4.1 機能要件
- RDP セッションを HTTPS でカプセル化すること。
- クライアントからの接続は TLS で暗号化されること。
- 認証はローカルユーザーで実施できること。
- 接続ログを取得できること。
- Windows の `mstsc.exe` からの接続に対応すること。
- 同時接続数は 2 を想定する。
- 接続対象の端末数は 2 台とする。

### 4.2 非機能要件
- 24/7 稼働を前提とする。
- 可用性は 99.9% を目標とする。
- 設定とログは監査に耐える形式で保管する。

### 4.3 ネットワーク要件
- インバウンドは TCP 443 のみを公開する。
- アウトバウンドは内部 RDP サーバー向け TCP 3389 を許可する。
- 企業プロキシ経由でも TCP 443 で外部アクセスできること。
- RD の対象 PC は Ubuntu と Tailscale で接続されていること。

## 5. 方式
- リバースプロキシ (例: Nginx) 上で TLS 終端を行う。
- RDP over HTTPS を提供するゲートウェイソフトウェアを導入する。
- RD の対象 Ubuntu には `xrdp` を導入する。
- Ubuntu 側は環境を汚さないため、可能な限り全てのコンポーネントを Docker で提供する。

## 6. セキュリティ
- TLS 1.2 以上を必須とする。
- 証明書は Let's Encrypt を利用し、Cloudflare の DNS-01 を用いて取得する。
- 管理アクセスは管理用ネットワークに限定する。

## 7. 監視・運用
- 接続成功/失敗のログを収集する。
- 証明書期限の監視を行う。
- CPU、メモリ、同時接続数を監視対象とする。

## 8. 未確定事項
- 利用するゲートウェイソフトウェアの選定。
- 冗長化方式 (アクティブ/スタンバイ など)。

## 9. 事前準備物
- Ubuntu LTS のゲートウェイ用サーバー 1 台。
- Ubuntu LTS の RD 対象サーバー 2 台。
- Cloudflare で管理するドメインと DNS ゾーン。
- Cloudflare API トークン (DNS 編集権限)。
- TCP 443 が到達可能なグローバル IP と適切な FW/ポート開放。
- Tailscale のアカウントと認証キー。

## 10. 必要パッケージ/コンポーネント
### 10.1 ゲートウェイ側 (ホスト)
- Docker Engine。
- Docker Compose プラグイン。
- Tailscale クライアント (ホスト導入、またはコンテナでの運用)。

### 10.2 ゲートウェイ側 (コンテナ)
- リバースプロキシ: `nginx` もしくは同等のコンテナ。
- RDP over HTTPS ゲートウェイ: 選定後に確定 (例: FreeRDP `rdpgw` など)。
- ACME クライアント: `certbot` + `certbot-dns-cloudflare` (Cloudflare DNS-01 対応)。

### 10.3 RD 対象 Ubuntu 側
- `xrdp`。
- `xorgxrdp`。
- Tailscale クライアント。
